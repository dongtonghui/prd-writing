# PRD Writing Skill

从会议纪要、用户故事或功能需求出发，撰写完整的产品需求文档（PRD）。覆盖功能模块拆解、看板任务分发、交叉评审、整合输出、UI Demo设计全流程。

---

## 触发条件

### 适用场景
用户输入包含以下任一关键词或场景：
- "写PRD" / "产品需求文档" / "PRD撰写"
- "功能模块拆解" / "模块拆解"
- "会议纪要转PRD" / "会议纪要" + "产品"
- "需求文档" / "feature spec" / "product requirement"
- 用户提供会议纪要的文本内容
- 用户提供用户故事（User Story）列表
- 用户描述产品功能需求，需要结构化文档输出

### 不适用场景
以下场景**不应使用**本skill，应使用其他skill：

| 场景 | 原因 | 应使用的Skill |
|------|------|-------------|
| 用户已有PRD，需要输出开发代码 | 本skill只到PRD文档，不涉及代码实现 | `writing-plans` / `subagent-driven-development` |
| 用户需要修复现有代码bug | 本skill不处理代码调试 | `systematic-debugging` |
| 用户要求直接写代码不做需求分析 | 本skill强制先做需求分析 | `writing-plans`（直接实施计划） |
| 用户只需要简单页面原型，无需求文档 | 本skill overhead 过大 | `claude-design`（直接生成HTML） |
| 用户要求做竞品市场调研 | 本skill聚焦需求文档，调研只是输入 | `kanban-worker` + `market-research` profile |

---

## 核心工作流

```
输入分析 → 模块拆解 → 看板分发 → 并行撰写 → 交叉评审 → 整合输出 → UI Demo设计
```

### Phase 1: 输入分析（Input Analysis）

**目标**：理解需求范围，识别信息缺口。

#### Step 1.1: 读取输入
- 若用户提供会议纪要/需求文本 → 直接分析
- 若用户提供文件路径 → 读取文件内容
- 若用户仅给一句话需求（如"做一个AI助手"）→ **进入信息补全流程**（见下方「需求模糊时的处理」）

#### Step 1.2: 提取关键信息
从输入中提取以下要素：

| 要素 | 说明 | 缺失时的处理 |
|------|------|-------------|
| 产品名称 | APP/系统名称 | 使用用户提到的名称，或标注「待确认」 |
| 目标用户 | B2B/B2C/内部用户 | 标注「待确认」 |
| 核心功能 | 主要功能点列表 | 从会议纪要逐条提取 |
| 业务规则 | 定价、权限、流程规则 | 逐条提取并标注优先级 |
| 时间约束 | 上线时间、里程碑 | 标注「待确认」 |
| 技术约束 | 平台（iOS/Android/Web）、技术栈 | 标注「待确认」 |
| 竞品参考 | 是否有对标产品 | 标注「无」 |

#### Step 1.3: 信息缺口检查

若以下**关键要素缺失 ≥2 项**，必须暂停并询问用户：
- 产品名称
- 核心功能（无任何功能描述）
- 目标用户

🔴 **CHECKPOINT · STOP**

> 信息缺口 ≥2 项时，向用户展示缺失清单，询问补充。示例：
> "检测到以下信息缺失，需要您补充：
> 1. 目标用户群体（B2B/B2C？）
> 2. 技术平台（iOS/Android/Web/小程序？）
> 3. 预期上线时间？
>
> 补充后可继续生成PRD，或基于合理假设直接开始（标注假设内容）。"

### Phase 2: 功能模块拆解（Module Decomposition）

**目标**：将需求拆分为独立、可并行的功能模块。

#### Step 2.1: 一级模块划分

基于需求内容，按以下维度划分一级模块：

| 维度 | 适用场景 | 示例 |
|------|---------|------|
| 用户角色 | 多角色权限差异大 | 管理员/普通用户/访客 |
| 功能域 | 功能间耦合度低 | 会员体系/支付系统/内容管理 |
| 业务流程 | 流程阶段清晰 | 注册→认证→下单→售后 |
| 页面/界面 | 前端驱动型产品 | 首页/详情页/个人中心 |

**原则**：
- 每个模块边界清晰，模块间通过明确接口交互
- 模块数量建议 3-8 个（过少则拆解不足，过多则管理 overhead 过大）
- 每个模块必须包含：模块名称、核心功能、输入、输出、依赖模块

#### Step 2.2: 二级功能点拆解

对每个一级模块，拆解为具体功能点：

```markdown
### 模块X: [模块名称]

**核心功能**: [一句话描述]
**输入**: [需要什么数据/触发条件]
**输出**: [产生什么结果/数据]
**依赖**: [依赖的其他模块]

#### 功能点清单
| 功能点 | 优先级 | 复杂度 | 备注 |
|--------|--------|--------|------|
| X.1 xxx | P0/P1/P2 | 高/中/低 | |
| X.2 xxx | P0/P1/P2 | 高/中/低 | |
```

#### Step 2.3: 数据模型初步识别

识别跨模块的共享数据实体：

| 实体 | 属性 | 所属模块 | 被哪些模块使用 |
|------|------|---------|--------------|
| User | id, name, role, status | 用户模块 | 所有模块 |
| Order | id, user_id, amount, status | 订单模块 | 支付、物流 |

### Phase 3: 看板任务分发（Kanban Dispatch）

**目标**：将模块拆解为可并行的看板任务，创建任务记录并分发给子agent执行。

**本Phase的边界**：
- ✅ 做：创建任务、定义任务内容、确认任务清单
- ❌ 不做：实际撰写PRD内容（那是Phase 4的工作）

#### Step 3.1: 创建看板任务

每个功能模块对应一个看板任务：

```json
{
  "task_id": "prd-[模块名]-[序号]",
  "title": "PRD: [模块名称] 需求文档撰写",
  "profile": "final-synthesis",
  "content": "基于以下需求，撰写[模块名称]的详细PRD子文档...",
  "dependencies": ["依赖的任务ID"]
}
```

**任务分配原则**：
- 无依赖的模块 → 立即分发，并行执行
- 有依赖的模块 → 等待依赖完成后分发
- 数据模型/全局规范 → 优先分发（其他模块依赖它）

#### Step 3.2: 子agent执行

使用 `delegate_task` 或 `kanban-worker` 分发任务：

- **profile**: `final-synthesis`（综合撰写）或 `market-research`（调研型模块）
- **context**: 完整的模块定义 + 全局约束 + 输出格式要求 + **模板内容（关键章节结构直接内联，不依赖文件路径）**
- **toolsets**: `['web', 'file']`（如需调研）或 `['file']`（纯撰写）

**⚠️ 子agent隔离环境警告**: 子agent无法访问主agent的skill目录文件（如`~/.hermes/skills/prd-writing/templates/`）。必须在context中直接提供：
1. 11章节结构清单（复制Step 4.1的markdown结构）
2. 数据模型字段要求（字段/类型/必填/默认值/说明）
3. 业务规则格式（规则ID/规则描述/触发条件/处理逻辑/优先级）
4. 异常处理格式（异常ID/场景/触发条件/系统处理/用户提示/日志记录）

**若子agent连续失败**: 按异常表「子agent超时/失败」处理 → 主agent直接撰写。

**子agent失败Fallback SOP（4步）**:
1. 记录失败任务ID、错误信息、已尝试次数
2. 重试1次（相同context，检查是否因网络/超时导致）
3. 仍失败 → **主agent直接撰写该模块**，不阻塞整体进度
4. 在最终PRD「已知问题」章节标注：`[fallback: 子agent失败，主agent撰写] + 失败原因摘要`

🔴 **CHECKPOINT · STOP**

> 展示看板任务清单，确认后再分发：
> "即将创建以下看板任务并并行执行：
> 1. PRD: 会员体系模块
> 2. PRD: 虚拟货币体系模块
> 3. PRD: 支付与退款模块
> ...
> 确认后分发执行？"

---

### Phase 4: 并行撰写（Parallel Writing）

**目标**：各子agent独立完成模块PRD撰写。

**本Phase的边界**：
- ✅ 做：子agent执行撰写、收集输出、质量检查
- ❌ 不做：模块拆解（Phase 2已完成）、最终整合（Phase 6的工作）

#### Step 4.1: 子agent输出格式

每个子agent必须输出以下结构的PRD子文档：

```markdown
# [模块名称] PRD

## 1. 功能概述
## 2. 用户故事
## 3. 功能流程
## 4. 页面/界面设计
## 5. 数据模型
## 6. API接口（如需要）
## 7. 业务规则
## 8. 异常处理
## 9. 验收标准
## 10. 依赖与风险
## 11. 附录
```

> **模板获取方式**（按优先级）：
> 1. 优先读取 `templates/prd-module-template.md` 获取完整模板（含字段说明和示例）
> 2. 若模板文件不可访问（子agent隔离环境），使用上方11章节结构直接输出
> 3. 子agent应填充内容，不改动11章节结构

**重要**: 子agent运行在隔离上下文，可能无法访问skill目录文件。主agent在分发任务时，应将模板内容或关键格式要求直接写入context，而非仅提供文件路径。

#### Step 4.2: 全局约束传递

向每个子agent传递以下全局约束：
- 产品名称、版本、目标用户
- 技术平台约束
- 术语表（统一用词）
- 与其他模块的接口约定

### Phase 5: 交叉评审（Cross Review）—— 三维度并行评审

**目标**：通过3个独立评审维度，全面发现模块间遗漏、冲突和方案缺陷。

**设计依据**：单一 `logic-review` 评审维度有限，易遗漏流程断点、方案不合理等问题。三维度并行评审借鉴 Darwin Skill 的「独立子agent评分」机制，每个维度由独立子agent盲评，主agent汇总裁决。

#### Step 5.1: 整合初稿

收集所有子agent输出，整合为单一文档：

```markdown
# [产品名称] PRD v1.0

## 目录
## 1. 产品概述
## 2. 功能架构图
## 3. 模块PRD（按模块组织）
## 4. 数据模型总览
## 5. 全局业务规则
## 6. 非功能需求
## 7. 附录（术语表、竞品分析等）
```

#### Step 5.2: 三维度并行评审

同时spawn **3个独立子agent**，每个agent只聚焦一个评审维度，彼此隔离、互不干扰。

**⚠️ 隔离原则**：
- 3个子agent必须在**独立上下文**中运行，不能看到彼此的评审结果
- 主agent等3个agent全部返回后，再汇总交叉比对
- 若子agent不可用（超时/资源限制），主agent直接执行该维度的评审 checklist

---

##### 维度A：流程图完整性评审（Flow Completeness Review）

**评审agent**: `logic-review` profile（或主agent fallback）
**聚焦目标**：所有功能流程是否完整、无断点、无遗漏分支

**评审Checklist**：

| 检查项 | 检查方法 | 不通过标准 |
|--------|---------|-----------|
| 主流程完整性 | 检查每个模块是否有Happy Path定义 | 缺失主流程 = 严重缺陷 |
| 分支覆盖度 | 检查所有条件分支是否都有处理路径 | 有if无else = 缺陷 |
| 异常流程覆盖 | 检查所有异常场景是否有处理 | 异常场景缺失 ≥2 处 = 严重缺陷 |
| 页面流程闭环 | 检查用户操作流程是否完整（入口→操作→结果→出口） | 流程断点 = 缺陷 |
| 跨模块流程衔接 | 检查模块间调用链是否连续 | A模块输出 ≠ B模块输入 = 缺陷 |
| 状态机完整性 | 检查实体状态转换是否覆盖所有可能 | 状态转换缺失 = 缺陷 |

**输出格式**：
```markdown
## 流程图完整性评审报告

### 严重缺陷（阻塞）
| # | 位置 | 问题 | 建议修复 |
|---|------|------|---------|
| FC-001 | [模块名/章节] | [问题描述] | [修复建议] |

### 一般缺陷（建议修复）
| # | 位置 | 问题 | 建议修复 |
|---|------|------|---------|
| FC-002 | [模块名/章节] | [问题描述] | [修复建议] |

### 通过项
- [检查项名称]: 通过 / 备注

### 评分
- 流程完整性: [0-10] / 10
- 分支覆盖度: [0-10] / 10
- 异常处理覆盖: [0-10] / 10
- **维度总分**: [0-30] / 30
```

---

##### 维度B：逻辑一致性评审（Logic Consistency Review）

**评审agent**: `logic-review` profile（或主agent fallback）
**聚焦目标**：模块间数据、术语、权限、接口是否一致，有无矛盾

**评审Checklist**：

| 检查项 | 检查方法 | 不通过标准 |
|--------|---------|-----------|
| 术语统一 | 搜索同一概念的不同表述 | 同一概念 ≥2 种表述 = 缺陷 |
| 数据模型冲突 | 对比各模块的数据定义（字段名/类型/约束） | 同一实体定义不一致 = 严重缺陷 |
| 接口一致性 | 检查模块间调用约定（参数名/类型/返回值） | 接口不匹配 = 严重缺陷 |
| 权限逻辑一致 | 检查各模块的权限判断规则 | 同一角色权限不一致 = 缺陷 |
| 业务规则冲突 | 检查跨模块业务规则是否矛盾 | 规则矛盾 = 严重缺陷 |
| 数值/常量一致 | 检查重复出现的数值（如分页大小、超时时间） | 同一常量多值 = 缺陷 |

**输出格式**：
```markdown
## 逻辑一致性评审报告

### 严重缺陷（阻塞）
| # | 位置 | 问题 | 建议修复 |
|---|------|------|---------|
| LC-001 | [模块A vs 模块B] | [冲突描述] | [修复建议] |

### 一般缺陷（建议修复）
| # | 位置 | 问题 | 建议修复 |
|---|------|------|---------|
| LC-002 | [模块名/章节] | [问题描述] | [修复建议] |

### 通过项
- [检查项名称]: 通过 / 备注

### 评分
- 术语一致性: [0-10] / 10
- 数据模型一致性: [0-10] / 10
- 业务规则一致性: [0-10] / 10
- **维度总分**: [0-30] / 30
```

##### 维度C：方案合理性评审（Solution Rationality Review）

**评审agent**: `logic-review` profile（或主agent fallback）
**聚焦目标**：方案设计是否合理、可落地、符合产品目标

**评审Checklist**：

| 检查项 | 检查方法 | 不通过标准 |
|--------|---------|-----------|
| 方案与目标对齐 | 检查每个方案是否支撑业务目标 | 方案与目标无关 = 严重缺陷 |
| 复杂度合理性 | 评估方案复杂度 vs 需求复杂度 | 过度设计（复杂度>>需求）= 缺陷 |
| 技术可行性 | 评估方案在当前技术约束下是否可落地 | 明显不可行 = 严重缺陷 |
| 用户体验合理性 | 检查交互设计是否符合用户心智模型 | 反直觉交互无说明 = 缺陷 |
| 性能风险 | 检查是否有明显性能瓶颈设计 | 高风险设计无缓解措施 = 缺陷 |
| 安全合规 | 检查敏感操作是否有安全机制 | 缺少鉴权/加密/审计 = 严重缺陷 |
| 扩展性 | 检查方案是否预留合理扩展空间 | 完全无扩展性 = 建议改进 |

**输出格式**：
```markdown
## 方案合理性评审报告

### 严重缺陷（阻塞）
| # | 位置 | 问题 | 建议修复 |
|---|------|------|---------|
| SR-001 | [模块名/章节] | [问题描述] | [修复建议] |

### 一般缺陷（建议修复）
| # | 位置 | 问题 | 建议修复 |
|---|------|------|---------|
| SR-002 | [模块名/章节] | [问题描述] | [修复建议] |

### 通过项
- [检查项名称]: 通过 / 备注

### 评分
- 目标对齐度: [0-10] / 10
- 技术可行性: [0-10] / 10
- 用户体验合理性: [0-10] / 10
- **维度总分**: [0-30] / 30
```

#### Step 5.3: 评审执行与汇总

**执行方式**：

```python
# 并行dispatch 3个评审子agent
delegate_task(
    tasks=[
        {
            "goal": "对以下PRD进行【流程图完整性】评审",
            "context": "## 评审维度：流程图完整性\n## 评审Checklist: [上方维度A的6项checklist]\n## 整合后PRD全文: [PRD内容]\n## 要求: 按维度A输出格式输出评审报告，只关注流程完整性，不评审逻辑一致性和方案合理性",
            "toolsets": ["file"]
        },
        {
            "goal": "对以下PRD进行【逻辑一致性】评审",
            "context": "## 评审维度：逻辑一致性\n## 评审Checklist: [上方维度B的6项checklist]\n## 整合后PRD全文: [PRD内容]\n## 要求: 按维度B输出格式输出评审报告，只关注逻辑一致性，不评审流程完整性和方案合理性",
            "toolsets": ["file"]
        },
        {
            "goal": "对以下PRD进行【方案合理性】评审",
            "context": "## 评审维度：方案合理性\n## 评审Checklist: [上方维度C的7项checklist]\n## 整合后PRD全文: [PRD内容]\n## 要求: 按维度C输出格式输出评审报告，只关注方案合理性，不评审流程完整性和逻辑一致性",
            "toolsets": ["file"]
        }
    ]
)
```

**汇总裁决**：

主agent收到3份评审报告后，执行以下裁决流程：

```
1. 提取所有【严重缺陷】（阻塞项）
   → 汇总为「必须修复清单」
   → 按模块分组，标注涉及维度（FC/LC/SR）

2. 提取所有【一般缺陷】（建议项）
   → 汇总为「建议修复清单」
   → 按优先级排序（影响范围大 > 实现成本低）

3. 交叉比对：
   → 同一位置被多个维度同时报缺陷 → 优先级提升
   → 维度间结论矛盾 → 主agent人工裁决（标注裁决理由）

4. 生成「评审汇总报告」:
```

**评审汇总报告格式**：

```markdown
# PRD交叉评审汇总报告

## 评审概览
| 维度 | 评审agent | 维度总分 | 状态 |
|------|----------|---------|------|
| 流程图完整性 | reviewer-A | [0-30] | ✓通过 / ⚠️有缺陷 / ✗不通过 |
| 逻辑一致性 | reviewer-B | [0-30] | ✓通过 / ⚠️有缺陷 / ✗不通过 |
| 方案合理性 | reviewer-C | [0-30] | ✓通过 / ⚠️有缺陷 / ✗不通过 |
| **综合评分** | - | **[0-90]** | - |

## 必须修复（阻塞项）
| # | 维度 | 位置 | 问题 | 修复建议 | 涉及模块 |
|---|------|------|------|---------|---------|
| 1 | FC | 支付模块/3.1 | 缺少支付失败后的回滚流程 | 补充异常流程EX-005 | 支付、订单 |
| 2 | LC | 用户模块/5.2 vs 订单模块/5.1 | User.status定义不一致（enum值不同） | 统一为active/inactive/suspended | 用户、订单 |
| 3 | SR | 虚拟货币模块/7.1 | 货币发行规则无上限控制，存在通胀风险 | 增加BR-010：日发行上限 | 虚拟货币 |

## 建议修复（非阻塞）
| # | 维度 | 位置 | 问题 | 修复建议 | 优先级 |
|---|------|------|------|---------|--------|
| 1 | FC | 会员模块/3.2 | 缺少会员过期后的降级流程 | 补充分支流程 | P1 |
| 2 | SR | 全局 | 未定义缓存策略，可能影响性能 | 在NFR中补充缓存要求 | P2 |

## 通过项（亮点）
- [维度]: [通过项描述]

## 裁决说明
- [如有维度间矛盾，说明主agent的裁决理由]
```

**通过标准**：
- ✅ **直接通过**：综合评分 ≥ 75 分，且无阻塞项
- ⚠️ **有条件通过**：综合评分 ≥ 60 分，阻塞项 ≤ 3 个（修复后通过）
- ✗ **不通过**：综合评分 < 60 分，或阻塞项 > 3 个（需重新设计）

🔴 **CHECKPOINT · STOP**

> 展示「评审汇总报告」，等用户确认修复优先级后再进入Phase 6。
> 
> "交叉评审完成，3个维度评审结果如下：
> - 流程图完整性: [分数] / 30 — [状态]
> - 逻辑一致性: [分数] / 30 — [状态]
> - 方案合理性: [分数] / 30 — [状态]
> - 综合评分: [分数] / 90
> 
> 发现 [N] 个必须修复项，[M] 个建议修复项。
> 是否按建议修复后进入整合输出？"

- 高优先级问题 → 立即修正
- 中优先级问题 → 标注「待确认」
- 低优先级问题 → 记录到「已知问题」

#### Step 6.2: 输出交付物

| 交付物 | 格式 | 保存路径 |

## 需求模糊时的处理（信息补全流程）

当用户输入过于模糊（如"做一个AI助手"），执行以下流程：

### Step A: 识别缺失信息

| 必需要素 | 缺失时的影响 |
|---------|-------------|
| 产品名称 | 无法统一术语 |
| 核心功能 | 无法拆解模块 |
| 目标用户 | 无法定义用户故事 |
| 技术平台 | 无法确定技术约束 |

### Step B: 提问策略

一次性列出所有缺失要素，不逐条追问：

> "为了写出准确的PRD，需要补充以下信息：
> 1. 产品名称和定位？
> 2. 目标用户是谁（B2B/B2C/内部）？
> 3. 核心功能有哪些？
> 4. 技术平台（iOS/Android/Web/小程序）？
> 5. 预期上线时间？
> 6. 是否有竞品参考？
>
> 您可以直接回复补充，或说'基于假设开始'（我会标注假设内容）。"

### Step C: 基于假设执行

若用户选择「基于假设开始」：
- 所有假设内容用 `[假设: xxx]` 标注
- 在PRD「已知问题」章节列出所有假设
- 建议用户后续确认假设内容

## 反例与黑名单（Don't Do This）

| # | 反模式 | 后果 | 正确做法 |
|---|--------|------|---------|
| 1 | **直接输出PRD不确认需求** | 产出与用户意图不符，返工成本高 | Phase 1必须做信息缺口检查 |
| 2 | **模块拆解过细（>10个模块）** | 管理 overhead 爆炸，看板任务过多 | 控制在3-8个模块，相关功能合并 |
| 3 | **跳过交叉评审直接整合** | 模块间数据模型冲突、术语不一致、方案缺陷未被发现 | Phase 5必须执行三维度并行评审 |
| 4 | **三维度评审不隔离** | 3个评审agent共享上下文，评审结论相互污染，失去独立评审意义 | 必须spawn独立子agent，彼此看不到对方结果 |
| 5 | **不标注假设直接写** | 用户误以为PRD基于确认事实 | 所有假设用 `[假设: xxx]` 显式标注 |
| 6 | **UI Demo覆盖全部页面** | 工作量过大，核心流程被稀释 | 只Demo核心流程页面（3-5个） |
| 7 | **子agent不带全局约束** | 各模块术语、数据定义不一致 | Phase 4必须传递术语表和接口约定；**context中直接内联模板，不依赖文件路径** |
| 8 | **子agent失败后无限重试** | 阻塞整体进度，浪费时间 | 重试1次后仍失败 → 主agent直接撰写（见Phase 3.2 Fallback SOP） |
| 9 | **评审问题不分类优先级** | 低优先级问题阻塞交付 | 分严重缺陷（阻塞）/一般缺陷（建议）两级，严重缺陷才阻塞 |
| 10 | **输出后不确认直接结束** | 用户可能有补充需求未表达 | Phase 6.3必须暂停确认 |

## 异常与边界条件

| 场景 | 触发条件 | 处理动作 |
|---|---|---|
| 用户输入非中文 | 检测到主要语言非中文 | 用用户语言输出PRD，但保留中文术语对照表 |
| 需求中途变更 | 用户在看板执行中补充新需求 | 记录到「需求变更日志」，评估是否影响已分发任务；影响大则暂停重新规划 |
| 子agent超时/失败 | delegate_task 返回错误 | 1. 记录失败任务ID和错误信息<br>2. 重试1次（相同context）<br>3. 仍失败 → 主agent直接撰写该模块，标注 `[fallback: 主agent撰写]`<br>4. 在「已知问题」记录子agent失败原因 |
| 模块间依赖循环 | A依赖B，B依赖A | 打破循环：提取公共依赖为独立模块，或标记为「需架构评审」 |
| 用户要求跳过某Phase | 用户说"直接写"或"跳过评审" | 告知风险（如"跳过评审可能导致模块不一致、流程断点、方案缺陷未被发现"），用户坚持则执行并标注 |
| PRD体积过大（>200KB） | 整合后文件过大 | 拆分为「主PRD+模块子文档」，主PRD保留摘要和链接 |
| 无看板系统可用 | kanban-worker 无法连接 | 退化为顺序执行：主agent逐模块撰写，不并行 |
| 子agent评审超时/失败 | 3个评审agent中部分失败 | 1. 记录失败维度 \u003cbr\u003e2. 主agent直接执行该维度的checklist \u003cbr\u003e3. 在评审汇总报告中标注 `[fallback: 主agent评审]` \u003cbr\u003e4. 继续汇总其他维度的评审结果 |

## 速查表（TL;DR）

```
用户给需求 → Phase 1 分析（缺口≥2则暂停询问）
         → Phase 2 拆模块（3-8个，含输入/输出/依赖）
         → Phase 3 看板分发（确认后并行）
         → Phase 4 子agent撰写（传全局约束）
         → Phase 5 交叉评审（三维度并行：流程完整性/逻辑一致性/方案合理性）
         → Phase 6 整合输出（修正+确认）
         → Phase 7 UI Demo（可选，核心页面）

红线：不确认需求直接写 / 模块>10个 / 跳过评审 / 评审不隔离 / 不标假设
```
