## 1. 评审模板速查

### 1.1 三维度评审 Agent Prompt 模板

#### 维度A：流程图完整性

```
你是一名专注于【流程图完整性】的PRD评审专家。

## 评审Checklist（6项）
1. 主流程完整性: 每个模块是否有Happy Path定义
2. 分支覆盖度: 所有条件分支是否都有处理路径（有if无else=缺陷）
3. 异常流程覆盖: 所有异常场景是否有处理
4. 页面流程闭环: 用户操作流程是否完整（入口→操作→结果→出口）
5. 跨模块流程衔接: 模块间调用链是否连续（A输出≠B输入=缺陷）
6. 状态机完整性: 实体状态转换是否覆盖所有可能

## 关键跨模块流程（根据产品调整）
- [模块A] → [模块B] → [模块C]
- [模块X] → [模块Y]（退款/失效等逆向流程）

## 输出格式
### 严重缺陷（阻塞）
| # | 位置 | 问题 | 建议修复 |

### 一般缺陷（建议修复）
| # | 位置 | 问题 | 建议修复 |

### 评分
- 流程完整性: [0-10]
- 分支覆盖度: [0-10]
- 异常处理覆盖: [0-10]
- 维度总分: [0-30]

**重要**: 只关注流程完整性，不评审逻辑一致性和方案合理性。
```

#### 维度B：逻辑一致性

```
你是一名专注于【逻辑一致性】的PRD评审专家。

## 评审Checklist（6项）
1. 术语统一: 同一概念的不同表述
2. 数据模型冲突: 字段名/类型/约束跨模块对比
3. 接口一致性: 模块间调用约定
4. 权限逻辑一致: 角色权限判断规则
5. 业务规则冲突: 跨模块规则矛盾
6. 数值/常量一致: 分页大小/超时时间/有效期等

## 输出格式（同维度A，编码前缀 LC-）
### 评分
- 术语一致性: [0-10]
- 数据模型一致性: [0-10]
- 业务规则一致性: [0-10]
- 维度总分: [0-30]

**重要**: 只关注逻辑一致性，不评审流程完整性和方案合理性。
```

```
你是一名专注于【方案合理性】的PRD评审专家。

## 评审Checklist（7项）

3. 技术可行性
```

## 2. 典型缺陷模式库

### 2.1 流程完整性类（FC）

| 缺陷模式 | 典型场景 | 检测方法 | 修复建议 |
|---------|---------|---------|---------|
| **有if无else** | 分支条件只覆盖正向路径 | 检查每个if是否有对应else/默认处理 | 补充else分支或标注「其他情况不处理」 |
| **跨模块链路断裂** | A模块输出未传递到B模块输入 | 追踪模块间接口调用 | 补充接口契约或事件通知 |
| **状态机死胡同** | 某状态无出口转换 | 绘制状态转换矩阵 | 补充转换路径或标注为终止状态 |
| **异常无兜底** | 异常处理后无用户提示/日志 | 检查每个异常的系统处理+用户提示+日志三要素 | 补充缺失要素 |
| **页面流程断点** | 用户操作后无明确结果页 | 检查入口→操作→结果→出口闭环 | 补充结果页或重定向 |

### 2.2 逻辑一致性类（LC）

| 缺陷模式 | 典型场景 | 检测方法 | 修复建议 |
|---------|---------|---------|---------|
| **枚举值冲突** | M1定义5种状态，M2定义4种 | 全局搜索状态枚举定义 | 统一枚举值，全模块强制使用 |
| **拼写不一致** | cancelled vs canceled | 全文搜索同一术语 | 统一拼写，建立术语表 |
| **字段类型冲突** | M3用TINYINT，M1用VARCHAR | 对比同一实体的字段定义 | 统一类型，建立数据字典 |
| **时效冲突** | M1:24h vs M5:48h | 对比同一流程的时效定义 | 统一时效，以最小公约数为准 |
| **接口契约缺失** | M2调用M1的接口但M1未定义 | 检查调用方接口在被调用方是否存在 | 补充接口定义或修正调用路径 |
| **记录冗余** | 两个模块都记录同一事件 | 检查重复数据实体 | 明确唯一数据源，其他模块引用 |

### 2.3 方案合理性类（SR）

| 缺陷模式 | 典型场景 | 检测方法 | 修复建议 |
|---------|---------|---------|---------|
| **合规风险** | 退款手续费涉嫌违反消费者保护法 | 对照当地法规检查 | 法务评审，调整规则 |
| **信任损害** | 自动取消用户已兑换订单 | 检查失效/过期处理逻辑 | 优先履约，无法履约则补偿 |
| **性能瓶颈** | 100万用户5分钟全表扫描 | 评估数据量+扫描方式 | 分片处理、索引优化、异步队列 |
| **目标不可达** | P99<5秒在跨渠道架构下不可行 | 对照技术约束评估 | 调整目标或架构 |
| **过度设计** | 为简单需求引入复杂状态机 | 对比需求复杂度与方案复杂度 | 简化方案，够用即可 |
| **缺乏模型支撑** | 95%自动审核覆盖率无数据支撑 | 要求提供历史数据或试点验证 | 先试点，再设定目标 |

## 3. MayGrove APP v3.0 实战案例

### 3.1 评审结果汇总

| 维度 | 评分 | 阻塞项 | 一般缺陷 |
|------|------|--------|---------|
| 流程完整性 | 20/30 | 8 | 6 |
| 逻辑一致性 | 14/30 | 8 | 20 |
| 方案合理性 | 20/30 | 6 | 12 |
| **综合** | **54/90** | **22** | **38** |

### 3.2 关键阻塞项与修正

| 编号 | 维度 | 问题 | 修正措施 |
|------|------|------|---------|
| LC-1 | 逻辑一致性 | 会员状态枚举冲突（M1 vs M2） | 统一为5种状态：active/paused/canceled/expired/refunded |
| LC-2 | 逻辑一致性 | 审核时效冲突（M1:24h vs M5:48h） | 统一为24小时 |
| LC-3 | 逻辑一致性 | 退款后权益处理矛盾 | 统一为"权益立即终止" |
| LC-4 | 逻辑一致性 | 会员等级编码不一致 | 统一VARCHAR编码 |
| LC-5 | 逻辑一致性 | 体验会员记录冗余 | 明确trial_record为唯一数据源 |
| LC-6 | 逻辑一致性 | M1↔M2接口契约缺失 | 补充activate/deactivate接口 |
| SR-1 | 方案合理性 | 退款手续费合规风险 | 标注待法务评审 |
| SR-2 | 方案合理性 | 批次失效清零冻结订单 | 修正为优先发货/积分补偿 |
| FC-1 | 流程完整性 | 退款缺少取消分支 | 补充取消流程 |
| FC-2 | 流程完整性 | 不续费分支无闭环 | 补充降级流程 |
| FC-3 | 流程完整性 | M2→M3/M4发放链路缺失 | 补充发放节点 |
| FC-4 | 流程完整性 | 退款扣回链路缺失 | 补充跨模块扣回流程 |

### 3.3 经验教训

1. **逻辑一致性是最容易出问题的维度** — 7个模块由不同子agent撰写，术语/枚举/字段极易不一致。必须在Phase 4全局约束中强制传递术语表和状态枚举。
2. **跨模块接口契约必须显式定义** — 不能依赖「子agent会自己推断」。主agent应在Phase 2模块拆解时就明确接口清单。
3. **方案合理性评审能发现合规风险** — 单一逻辑评审无法发现退款手续费涉嫌违法的问题，需要方案合理性维度的业务视角。
4. **评审后修正比想象中多** — 12个阻塞项全部修正后，综合评分从54提升到预估75+，说明评审投入产出比高。
5. **主PRD+模块子文档结构必要** — 7个模块总计296KB，超过200KB阈值。主PRD保留全局规范+摘要+评审结果，模块细节放子文档。

## 4. 复用检查清单

每次执行三维度评审后，对照以下清单确认：

- [ ] 3个评审agent在独立上下文中运行（无共享结果）
- [ ] 每个维度输出包含严重缺陷/一般缺陷/通过项/评分
- [ ] 阻塞项数量 ≤ 3 且综合评分 ≥ 60 才进入Phase 6
- [ ] 阻塞项修正后在主PRD中标注 `[vX.X 修正]`
- [ ] 待确认问题（PENDING_CONFIRMATION）单独列出，标注负责人和优先级
- [ ] 评审报告保存到 `/workspace/reports/prd-modules-vX/review-*.md`
- [ ] 若PRD体积 > 200KB，拆分为「主PRD + 模块子文档」
